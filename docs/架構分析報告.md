## 1. 專案結構總覽

### 整體架構
FlowCraft 是一個基於 **Vue 3 + Express** 的自動化工作流引擎，採用前後端分離架構，支援可視化拖拽式工作流設計、節點執行引擎、即時事件推送等功能。

## 3. 設計模式分析

### 核心設計模式

#### 1. Workflow Engine 模式
**實現位置**：`/server/src/execution/WorkflowEngine.ts`

**核心職責**：
- 工作流執行協調
- 依賴解析（DAG 拓撲排序）
- 節點生命週期管理
- 條件執行與跳過邏輯

**執行流程**：
```
1. 找出起始節點（沒有輸入邊且有輸出邊）
2. 區分觸發器和一般節點
3. 根據觸發器模式執行（Fallback vs Sequential）
4. 執行非觸發器起始節點
5. 依賴驅動執行（cascade execution）
6. 完成執行
```

**優點**：
- 清晰的執行流程
- 支援複雜的依賴關係
- 條件執行（減少無效計算）
- 快取機制（提升效能）

**問題**：
- 沒有處理循環依賴（可能導致死鎖）
- 沒有執行超時機制
- 錯誤處理簡單（一個節點失敗整個工作流停止）

#### 2. Node Registry 模式
**實現位置**：`/src/nodes/registry.ts`

**優點**：
- 聲明式節點定義
- 集中管理（易於查找）
- 支援版本管理

**問題**：
- 單一檔案 783 行，容易產生 Git 衝突
- 缺少節點驗證
- 沒有動態載入機制

#### 3. Executor 模式
**實現位置**：`/server/src/executor.ts` + `/server/src/executors/`

**優點**：
- 每個節點獨立實現（低耦合）
- 統一的 `emit` 介面
- 支援複雜邏輯

**問題**：
- `executor.ts` 的 switch case 是衝突熱點
- 缺少生命週期管理
- 沒有執行超時控制

#### 4. Event-Driven Architecture
**前端監聽**：
```typescript
socket.on('execution:start', ...)
socket.on('node:done', ...)
socket.on('node:log', ...)
```

**優點**：
- 即時 UI 更新
- 支援多客戶端同步觀察
- 解耦前後端

**問題**：
- 沒有事件重播機制
- 沒有執行歷史持久化
- WebSocket 斷線缺少優雅降級

## 4. 核心模組深度分析

### 4.1 WorkflowEngine（執行引擎）

**檔案路徑**：`/server/src/execution/WorkflowEngine.ts`（720 行）

#### 依賴解析算法
```typescript
private canExecuteNode(nodeId: string): boolean {
  const incomingEdges = this.findIncomingEdges(nodeId)
  return incomingEdges.every(edge => this.completedNodes.has(edge.source))
}
```

**特點**：
- 簡單的依賴檢查（O(E)）
- 不支援循環依賴檢測

**建議改進**：使用拓撲排序預先檢測循環依賴

#### 條件執行邏輯
- 檢查所有輸入是否為 null
- 如果都是 null → 跳過執行
- 輸出 null 讓下游也跳過

**問題**：
- 邏輯分散在多處
- 難以擴展（例如：OR 條件）

#### 錯誤處理機制
- 一個節點失敗 → 整個工作流停止
- 沒有 retry 邏輯
- 沒有 fallback 機制

**建議改進**：節點層級錯誤策略（stop/continue/retry/fallback）

#### 觸發器輪詢機制
**兩種模式**：
1. **Fallback 模式**：依序嘗試，第一個有內容就停止
2. **Sequential 模式**：依序執行所有觸發器

**問題**：
- 沒有真正的定時輪詢
- State 檔案可能在多個工作流間衝突
- 缺少 trigger queue 管理

### 4.2 Node Registry（節點註冊中心）

**檔案路徑**：`/src/nodes/registry.ts`（783 行）

#### 節點分類
| 分類 | 節點數量 | 代表節點 |
|------|----------|----------|
| trigger | 7 | YouTube Monitor, Schedule, Manual Trigger |
| action | 3 | Send Telegram, Send Discord, HTTP Request |
| ai | 7 | Gemini, NotebookLM, Segment Mining |
| data | 6 | Google Sheets, Execution Logger |
| media | 5 | YouTube Subtitle, YouTube Thumbnail |
| logic | 6 | Filter, Delay, Code |

#### 擴展機制
- 自定義配置組件：`customConfig: 'YouTubeMonitorConfig'`
- 需手動註冊在 Editor.vue

**問題**：
- 自定義組件容易遺漏
- 沒有組件載入錯誤處理
- 沒有組件版本管理

#### 版本管理
- 每個節點有 `version` 欄位
- 但沒有實際的版本檢查邏輯
- 舊版工作流載入新版節點可能出錯

**建議改進**：自動遷移機制

## 5. 資料流分析

### 5.1 前後端通訊

#### HTTP API Endpoints
```
POST /api/execute              # 單一節點執行
POST /api/workflow/run         # 工作流執行
GET  /api/health               # 健康檢查
POST /api/auth/notebooklm      # NotebookLM 授權
GET  /api/settings/*           # 設定相關
GET  /api/youtube/*            # YouTube 相關
POST /api/workflows/sync       # 工作流同步
```

#### WebSocket 事件流
```
執行層級：
  execution:start
  execution:complete
  execution:failed

節點層級：
  node:start
  node:done
  node:error
  node:log
  node:progress
  node:skipped
```

### 5.2 工作流執行資料流

#### 節點間資料傳遞
```
YouTube Monitor (out-url, out-title)
    ↓
YouTube Subtitle (in-url) → (out-transcript)
    ↓
Segment Mining (in-source) → (out-result)
    ↓
Script Generator (in-source) → (out-script)
    ↓
Send Telegram (in-message)
```

#### 快取機制
**快取儲存**（前端）：
- 儲存在 Pinia store + localStorage
- 按 nodeId 索引

**快取使用**（後端）：
- 檢查 `node.data.useCachedResult`
- 有快取就跳過執行

**問題**：
- 快取無過期機制
- 快取鍵僅用 `nodeId`（可能衝突）
- 沒有容量限制

**建議改進**：
- 加入過期時間
- 快取鍵改為：workflowId + nodeId + configHash
- 實現 LRU 淘汰策略

## 7. 架構問題與風險

### 7.1 高優先級問題

#### 1. 缺少循環依賴檢測
**風險**：工作流可能死鎖

**範例**：
```
Node A → Node B
Node B → Node C
Node C → Node A  (循環！)
```

**解法**：執行前拓撲排序檢測循環

#### 2. 執行超時無控制
**風險**：
- NotebookLM 可能無限等待
- AI API 呼叫可能掛住
- 整個工作流卡死

**解法**：引擎層統一超時控制（Promise.race）

#### 3. 錯誤處理不足
**現狀**：一個節點失敗 → 整個工作流停止

**解法**：節點層級錯誤策略
```typescript
node.data.errorStrategy = {
  onError: 'retry',
  maxRetries: 3,
  retryDelay: 5000
}
```

#### 4. WebSocket 斷線無重播
**風險**：
- 用戶刷新頁面 → 執行歷史丟失
- WebSocket 斷線 → 事件丟失

**解法**：後端儲存執行歷史，前端重連時載入

### 7.2 中優先級問題

#### 5. Node Registry 單檔案過大
**問題**：
- 783 行，容易衝突
- 節點定義與實現分離

**解法**：節點定義與實現合併

#### 6. Trigger 系統不完整
**問題**：
- 定義了 `triggerType: 'cron'`，但沒有 scheduler
- Webhook Trigger 沒有接收器

**解法**：實現 Cron Scheduler

#### 7. 快取機制不健全
**解法**：見上文「快取機制」章節

#### 8. 缺少執行歷史管理
**解法**：ExecutionHistory Store + UI 頁面

### 7.3 低優先級問題

#### 9. 缺少單元測試
#### 10. TypeScript 類型覆蓋不足
#### 11. 沒有 API 文件
#### 12. 前端性能優化空間

## 9. 總結

### 整體評價

FlowCraft 是一個架構清晰、模組化良好的工作流自動化系統。

**技術優勢**：
- 前後端分離，職責清晰
- 可視化工作流，用戶體驗佳
- 模組化設計，易於擴展
- 即時反饋，執行進度清晰
- AI 整合完善，錯誤處理友善

**架構風險**：
- 缺少循環依賴檢測
- 執行超時無控制
- 錯誤處理簡單
- Trigger 系統不完整
- 快取機制不健全

**可擴展性**：
- 節點擴展容易
- 前端組件化清晰
- Registry 衝突風險需注意

**維護性**：
- 協作流程清晰
- 缺少測試
- TypeScript 覆蓋不足

### 適用場景

**最適合**：
- 個人/小團隊自動化工作流
- AI 內容生產 Pipeline
- 資料採集與處理
- 社群媒體自動化

**不適合**：
- 高可用性需求（缺少容錯機制）
- 大規模並行執行（缺少任務佇列）
- 企業級多租戶（缺少權限管理）
